当我们编写并运行“Hello World”的时候，电脑到底发生了什么？

本文是笔者在阅读CSAPP（《深入理解计算机系统》）一书时整理出来的。希望对大家有所帮助。

编写源代码

我们从源代码的编写开始。用任何一个你喜欢的文本编辑器，用C语言写出我们经典的Hello World程序。

保存上述文件为“hello.c”，我们就得到了源文件。
我们的源文件实际上是一个由”0”和”1”组成的位(bit)序列，每8个位组成一组，称为字节（byte)。源文件中的每一个字符，都由一个确定的字节所表示。
我们使用ASCII编码来讲述。关于ASCII、Unicode、UTF-8编码的区别，请参考这篇博客：http://www.cnblogs.com/kingstarspe/p/ASCII.html
举个例子，我们源文件中，第一个字符是”#”，它的ASCII码是”35”，表示成8位”0”和”1”就是“00010011”(高位在左)。计算机给每一个字符都采用上述方式进行编码，我们就可以得到这样一串数据：

hello.c的表示方法说明了一个基本的思想：系统中所有的信息——包括磁盘文件、存储器中的程序、存储器中存放的用户数据以及网络上传送的数据，都是由一串位标识的。 我们区分数据所表示的对象要根据上下文。比如，同一个字节序列，在不同的上下文中，可能表示一个整数、浮点数、字符串或者机器指令。

编译成可执行文件

从源文件转化为可执行目标文件是由编译器驱动程序完成的。以linux系统为例，我们可以这样做：

在这里，GCC编译器驱动程序读取源文件hello.c，并且把它翻译成一个可执行文件hello。这整个过程可分为以下几个阶段：

  • 预处理阶段：
    预处理器根据以字符“#”开头的命令，修改原始的程序。在我们的程序中，就是根据“#include ”命令，预处理器读取系统头文件”stdio.h”的内容，并将它直接插入到我们的程序中。这样就得到了另一个程序，这个程序通常以”.i”为扩展名保存。
  • 编译阶段： 编译器将文本文件”hello.i”翻译成另一个文本文件”hello.s”，它的内容是一个汇编程序。汇编程序中的每一条语句都以一种标准的文本格式确切的描述一条低级机器语言指令。
  • 汇编阶段：
    汇编器将”hello.s”翻译成机器语言指令，将这些指令打包成 可重定位目标程序(relocatable object program) 的格式，并将结果保存在目标文件”hello.o”中。”hello.o”文件是一个二进制文件，它的字节编码是机器语言指令而不是字符。

  • 链接阶段：
    我们最开始编写的源代码中，调用了一个”printf”函数，它是标准C库里面的函数。”printf”函数存在于一个名为”printf.o”的已经单独编译好的目标文件中。 链接器 的作用就是合并”printf.o”文件到”hello.o”文件中。结果的到”hello”文件，他是一个 可执行目标文件 ，可以被加载到内存中，由系统执行。

计算机执行程序

这是最复杂的一个步骤，也是最精秒的一个步骤。 我们首先看看我们的程序执行的结果是什么：

很简单，就是在shell(一个命令行解释器)中输入”./hello”，然后shell会判断这个命令第一个单词是否是内置的命令，如果不是，shell会假设这是一个可执行文件的名字，并且将它加载、运行，同时将结果显示在屏幕上。结束这个程序后，shell会输出新的提示符，等待下一条命令。
要了解这一过程中，计算机到底做了什么，首先我们要了解计算机的硬件组成。

系统的硬件组成

重新画了一张书上的图片：

简单介绍一下上述几个概念：

  • 总线
    总线式贯穿整个系统的一组电子管道。它携带信息字节并且在各个部件之间传递。总线被设计成传送定长的字节块，称为字(word)。
  • I/O设备
    I/O设备即 输入/输出设备 。上图包括四个I/O设备：鼠标、键盘、显示器、磁盘。每个设备都通过一个 控制器/适配器 与I/O总线相连。控制器和适配器的主要区别就是：控制器是封装在主板上的芯片组，适配器则是主板插槽上的卡。
  • 主存储器
    主存储器(主存)是一个临时存储设备，程序执行时，用来存放程序以及程序处理的数据。主存由一组 动态随机存取存储器(DRAM) 芯片组成的，可看成线性的字节数组，每个字节都有一个唯一的地址。
  • 处理器
    中央处理单元(CPU)简称处理器。是解释(或执行)存储在主存中指令的引擎。处理器的核心是一个字长的存储设备(或寄存器)，称为 程序计数器(PC) 。任何时刻，PC都指向主存中的某条机器语言指令。
  • ALU
    ALU即算数/逻辑单元。ALU的工作是计算新的数据和地址。
  • 寄存器文件
    寄存器文件是一个小的内存设备，由一些字长为1的寄存器组成，每个寄存器都有唯一的名字。

CPU在指令的要求下，可能会执行以下操作：

  • 加载 ：把一个字节或者一个字，从主存复制到寄存器，以覆盖寄存器原来的内容。
  • 存储 ：把一个字节或者一个字，从寄存器复制到主存的某个位置，以覆盖主存原来的内容。
  • 操作 ：把两个寄存器的内容复制到ALU，ALU对这两个字做算数操作，并将结果存放在一个寄存器中，以覆盖这个寄存器原来的内容。
  • 跳转 ：从指令本身抽取一个字，并将这个字复制到程序计数器(PC)中，以覆盖原来PC中的内容。

运行hello程序的过程发生了什么

我们回到shell输入”./hello”这个步骤。
“./hello”这个字符串是我们从I/O设备——键盘——输入的。shell首先将字符逐一读入到寄存器，然后将他们放到存储器中。这一过程如下图：

绿色的”1”号线路是从键盘的输入字符读取到寄存器；
粉色的”2”号线是从寄存器的字符放到主存的过程，此时主存存有”hello”这个字符串。

当我们在键盘上敲击”enter”键时，shell就知道我们已经结束了命令的输入。然后shell会运行一系列指令来加载我们的可执行文件”hello”，然后将”hello”可执行目标文件中的代码和数据从磁盘复制到主存，这里的数据包括屏幕将要输出的”hello, world\n”字符串。这个过程如下图：

红色的”3”号线是从磁盘复制程序和数据到主存的过程，此时主存含有hello程序和”hello, world\n”字符串。

一旦目标文件hello的代码和数据被加载到主存，CPU就开始执行hello程序的main程序中的机器语言指令，这些指令将”hello, world\n”字符串中的字节，从主存复制到寄存器文件，再将寄存器文件中复制到显示设备，最终显示到屏幕上。这个过程如下图：

紫色的”4”号线是主存中的指令加载到寄存器文件；
灰色的”5”号线是将寄存器文件中的内容复制到显示设备，最终打印到屏幕上。

Android中的回调举例分析

网上也有不少讲回调的博客，但是大多数都是谈该怎么做，却没有说清楚为什么需要这样。本文将从一个初学者的角度出来，来描述一个从不会到会的成长过程。

怎么理解“回调”

先说说我的理解：在一个逻辑上完备但是某些细节有待填充的项目，你去实现那些特意暴露出来的接口，这就是回调的目的和过程。
一开始可能会不理解这句话，等你看完这篇博客，回过头看看，你大概就能理解了。不妨先往下看。

为何要用回调？

回调的应用很广泛，在Android开发中也是处处可见。本文不打算从系统的角度（其实也是自身能力所限）来讲为何需要回调，仅从普通开发需要来说明。
试想这样一个情景：你在屏幕上绘制了一个按钮，你需要实现这样的功能——长按按钮，弹出一个对话框，你可以在这里设置按钮的颜色以及大小等属性，点击“保存”按钮之后，
按钮的属性作出相应的改变。
这个时候，你会怎么实现该功能呢？
我们来分下下这个问题。这个问题的关键点在于：我需要在对话框进行属性配置，但是效果却需要体现在按钮身上，那么他们两者之间是不是需要某种确定的联系呢？
答案是肯定的。
我们这样想：效果要体现在按钮上，那么这种联系的具体实现肯定在按钮这一边;但是按钮这一边并不知道逻辑上如何实现啊？那就是说，应该在
对话框这边实现逻辑，并且暴露某些接口让按钮一边去实现，而按钮不需要知道逻辑是怎么样的，它只需要实现对话框暴露的接口。
这样也就是说，对话框是主动方，按钮是被动方：我对话框让你干嘛，你按钮就去干嘛。
至此，他们之间的联系已经建立起来了！那么谁最适合去充当这个联系人的角色呢？当然是接口(interface)!!!
可能你还是不太明白，先别急，不妨看看代码再说。

如何实现回调？

这里我们来写一个实例，让你感受一下到底该如何实现。
我们假设，在一个Activity里面，添加一个按钮，长按按钮弹出对话框。
首先，写出我们基本的Activity： 

public class MainActivity extends AppCompatActivity implements MyDialogListener{

    private MyDialog mDialog;
    private Button button;

    @Override
    public void onCreate(Bundle savedInstanceState){
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        button = (Button) findViewById(R.id.main_button);
        mDialog = new MyDialog(this, this);
        button.setOnLongClickListener(new View.OnLongClickListener{
            @Override
            public void onLongClick(View view){
                mDialog.show();
            }
        });

        button.setLayoutParams(new FrameLayout.LayoutParams(
            ViewGroup.LayoutParams.WRAP_CONTENT, ViewGroup.LayoutParams.WRAP_CONTENT));
    }

    @Override
    public void onSave(Map<String, String> map){
        //...
    }
}

上面的代码很简单，MainActivity有一个按钮，还拥有一个对话框成员变量。按钮长按的时候，弹出对话框。
接下来要实现对话框MyDialog。
除此之外，MainActivity实现了一个MyDialogListener接口，该接口就是我们的联系人。

public class MyDialog extends Dialog{

    private MyDialogListener mListener;
    private int colour;

    public MyDialog(Context context, MyDialogListener listener){
        super(context);
        setContentView(R.layout.dialog);

        //持有MyDialogListener的引用
        mListener = listener;

        Spinner color = (Spinner) findViewById(R.id.spinner_color);         //选择颜色
        EditText width = (EditText) findViewById(R.id.edit_text_width);     //编辑宽度
        EditText height = (EditText) findViewById(R.id.edit_text_height);   //编辑长度

        Button save = (Button) findViewById(R.id.button_save);              //保存按钮

        color.setOnItemSelectedListener(new AdapterView.OnItemSelectedListener() {
            @Override
            public void onItemSelected(AdapterView<?> adapterView, View view, int i, long l) {
                colour = i;
            }

            @Override
            public void onNothingSelected(AdapterView<?> adapterView) {

            }
        });
        save.setOnClickListener(new View.OnClickListener{
            @Override
            public void onClick(View view){
                Map<String, String> map = new HashMap<>();
                map.put("color", colour+"");
                map.put("width", width.getText().toString());
                map.put("height", height.getText().toString());

                //调用MyDialogListener的onSave()方法
                //至此说明，逻辑上以及完成了，至于onSave()方法是怎么实现的，交给MyDialogListener就行了
                mListener.onSave(map);
            }
        })

    }

    //定义一个接口，作为MyDialog与外界的联系人
    public interface MyDialogListener{
        void onSave(Map<String, String> map);
    }
}

至此，MyDialog的逻辑已经完全实现了，只要实现了MyDialogListener接口的类，都可以使用MyDialog来实现上述操作，具体怎么
实现，就要看自身(实现了MyDialogListener接口的类)如何实现onSave()方法了。

我们把MainActivity覆盖的onSave()方法具体实现如下： 


@Override
public void onSave(Map<String, String> map){
    int color  = Integer.pearseInt(map.get("color"));
    int width = Integer.pearseInt(map.get("width"));
    int height = Integer.pearseInt(map.get("height"));

    button.setBackgroundColor(color);
    FramLayout.LayoutParams params = button.getLayoutParams();
    params.width = width;
    params.height = height;
    button.setLayoutParams(params);

}

Android MVP模式

MVP即Model、View和Presenter。
使用MVP的模式可以使得Model和View层解耦，他们之间的交互通过Presenter来进行。

可以这样看：

  • Model层负责具体业务逻辑的实现，比如网络接发数据，数据库操作等等
  • View层负责与用户的交互以及界面的更新
  • Presenter层负责逻辑分发，根据View层的不同命令，与Model交互，并将结果返回给View。

这样就达到了View与Model解耦的效果。
以下根据我的实际项目来讲解MVP模式。这个项目刚刚开始，远还没有完成，但是我一开始就是根据MVP模式来设计的，因此很有参考意义。
当然我也是第一次在项目中使用MVP模式，有错误的地方还请指出。项目开源在Github：[EmbeddedTool]:https://github.com/StupidL/EmbeddedTool

项目简介

这个项目主要是为了与嵌入式设备进行交互，可以很容易地通过控件组合来完成各种功能。与嵌入式设备的数据传输通过蓝牙或者wifi。
目前已经完成了一下一个基本控件：

  • 发送类型的按钮，可以直接发送某种命令或者与其他控件结合使用
  • 接收类型的按钮，可以直接接收数据或者与其他控件组合使用
  • 文本框，用来显示发送的数据或者接收的数据，可以绑定两种类型的按钮一起工作
  • 编辑框，发送某些数据，可以绑定一个发送类型的按钮一起工作

以下一按钮功能的实现为例。

项目架构

Model层：

  • 定义一个INetModel接口，声明网络接收数据的各种操作
  • 定义一个NetModelImpl实现类，实现上述接口 
public interface INetModel{
    void sendDataOverButton(String s);
    String receiveDataOverButton();
}

public NetModelImpl implements INetModel{
    @Override
    public void sendDataOverButton(String s){
        //...
    }
    @Override
    public String receiveDataOverButton(){
        //...
    }
}

View层：

  • 定义一个接口，声明用户交互和界面更新所需要的操作
  • 在对应的Activity(不一定是Activity)实现上述接口 
public interface IMainView{
    void addButton();
    void removeButton(Button button);
}

public class MainActivity extends AppCompatActivity implements IMainView{
    //...life circle methods

    @Override
    public void addButton(){
        //...
    }

    @Override
    public void removeButton(Button button){
        //...
    }
}

以上两层写好之后，问题来了：我从Model层接收的数据该如何显示到View层？我要把View层的数据通过网络发送出去该怎么办？
这正是Presenter该做的事情！

Presenter层：

  • 定义一个MainPresenter类，来分发各种逻辑 
public class MainPresenter{
    public void sendData(String s){
        //...
    }
    public String receiveData(){
        //...
    }
    public void addButton(){
        //...
    }
    public void removeButton(Button button){
        //...
    }
}

以上只是一个框架，那么Presenter里面的各种方法该如何实现呢？换句话说，presenter该如何与model和view联系起来呢？
具体的做法是：presenter里面持有view和model两个成员变量！并且在构造器重初始化这两个成员！！！
所以，我们的presenter就变成了以下这个样子： 

public class MainPresenter{
    private IMainView iMainView;
    private INetModel inetModel;
    public MainPresenter(IMainView view){
        iMainView = view;
        inetModel = new NetModelImpl();
    }
    public void sendData(String s){
        inetModel.sendDataOverButton(s);
        //...other operations
    }
    public String receiveData(){
        //...other operations
        return inetModel.receiveDataOverButton();
    }
    public void addButton(){
        iMainView.addButton();
        //...other operations
    }
    public void removeButton(Button button){
        iMainView.removeButton(button);
        //...other operations
    }
}

现在，新的问题来了：我没看到View层和Presenter的具体联系啊，他们怎么进行交互呢？
我们知道View与Presenter的交互是双向的。或许你已经发现了，在Presenter的构造器里，有一个IMainView参数。
如果你还记得我们的MainActivity实现了IMainView接口，那你就懂了！也就是说，我们的Presenter持有MainActivity的
一个引用！这就意味着Presenter可以访问View！
那么View该如何访问Presenter呢？同样的办法，将presenter作为View的一个成员变量就可以了！
因此我们的Activity变成了下面这个样子： 

public class MainActivity extends AppCompatActivity implements IMainView{

    private MainPresenter mPresenter;

    @Override
    public void onCreate(Bundle savedInstanceState){
        super.onCreate(savedInstanceState);

        mPresenter = new MainPresenter(this);

        //...

        //add a button by another button.
        Button button = new Button(this);
        button.setOnClickListener(new OnClickListener{
            @Override
            public void onClick(View view){
                //just show that you can use presenter to finish some jobs rather than view itself.
                mPresenter.addButton();
                mPresenter.sendDataOverButton("...");
            }
        });

        //remove a button by another button.
        Button button2 = new Button(this);
        button2.setOnClickListener(new OnClickListener{
            @Override
            public void onClick(View view){
                mPresenter.removeButton(button);    //arg button should be found first.
            }
        });
    }

    //...other life circle methods

    @Override

Java Interface

在Java中，接口(interface)和内部类为我们提供了一种将接口与实现分离的更加结构化的方法。

抽象类与抽象方法

包含抽象方法的类叫做抽象类。如果一个类包含一个或者多个抽象方法，则该类必须限定为抽象的。
抽象类其实起着这样一个作用——通用接口。不同的子类可以用不同的方式表示此接口。这和我们讨论的接口是不是很像呢？这也是为什么在说接口(interface)之前要说抽象类的原因。
如果想从一个抽象类继承，并且创建该新类的对象，则必须为基类中的所有抽象方法提供方法定义。这一点与C++是一样的。如果不这么做，那么导出类也是一个抽象类，并且编译器会强制我们使用
abstract关键字来限定这个类。
创建抽象类和抽象方法非常有用，因为它们可以使累的抽象性明确起来，并且告诉用户和编译器打算怎么来使用它们。

你真的理解了接口吗？

abstract关键字允许人们在类中创建一个或者多个没有任何定义的方法——提供了接口部分，但是没有提供具体的实现，这些是此类的导出类完成的。
interface关键字使得抽象的概念更向前迈进了一步。interface关键字创建一个完全抽象的类：它根本就没有提供任何具体实现。它允许创建者定义方法和签名，但是没有任何方法体。
（注：Java 8 支持在interface中包含方法体，这是Java 8的新特性之一。更多新特性，请自行查看Java 8 release note）
一个接口表示：“所有实现了该特定接口的类看起来都像这样”。
interface不仅仅是一个极度抽象的类，它还允许我们创建一个能够被向上转型为多种基类的类型。
创建一个接口很简单，用interface关键字代替class关键字即可。例如： 

interface SampleInterface{}

要让一个类遵循某个接口（或者一组接口），需要使用implements关键字，它表示：“interface知识外貌，我现在要说明它是如何工作的”。也就是实现interface中声明的方法。例如： 

class SampleClass implements SampleInterface{}

值得注意的是：

  • 接口是隐式抽象的，不需要abstract关键字修饰
  • 接口中的方法也是隐式抽象的，不需要abstract关键字修饰
  • 接口中的方法访问权限都是public

接口支持多重继承！

我们知道，Java中类是不支持多重继承的（C++支持多继承），但是接口是完全支持多继承的。一个类想“继承”多个接口，只需要在implements关键字后面，逐个写出接口名，用“，”隔开。例如： 

interface I1{ void method1();}  
interface I2{ void method2();}
interface I3{ void method3();}  
class A implements I1, I2, I3{
  void method1(){}
  void method2(){}
  void method3(){}
}

不知道你发现没有，这样的特性使得该类可以向上转型为任何一个接口类型，因为每一个接口都是一个独立的类型！ 这样的特性十分灵活！
总结一下使用接口的原因：

  • 为了能够向上转型为多个类型以及由此带来的灵活性
  • 防止客户端程序员创建该类的对象，并且确定这只是一个接口——这一点与抽象类相同。

那么，我们应该使用接口还是抽象类呢？
如果要创建不带任何方法定义和成员变量的基类，那么就应该选择接口而不是抽象类。

接口的扩展

说到扩展接口，第一反应就是通过继承现有的接口。比如这样： 


interface A {}  
interface B extends A{ }

这里的extends关键字是不是很熟悉？没错，它和非接口类的继承使用同样的关键字。我们知道一般的类extends后面只能接一个基类。但是 接口可以继承多个基类！
例如： 

interface A { void methodA(); }  
interface B { void methodB(); }  
interface C extends A, B { void methodC(); }

这样一来，如果有一个类实现接口C，实际上也实现了接口A和B——C继承了A和B的方法，类实现C就必须实现其所有方法！
很显然，接口C是一个扩展的接口！

又一个麻烦

不知道你想过没有，假设有这样一种情况：覆盖、实现和重载搅在了一起，该怎么办？
举个例子： 

interface A{ void f(); }  
interface B{ int f(int i); }  
interface C{ int f(); }  
class C1 { public int f(){ return 1;}}  
class C2 implements A, B{
  public void f(){}  
  public int f(int i){ return 1;} //overload  
}   
class C3 extends C1 implements B{
  public int f(int i){ return 1;} //overload  
}  
class C4 extends C1 implements C{
  public int f() { return 1; }
}

你觉得以上程序有什么错误吗？如果加上以下两行呢？ 

class C5 extends C implements A{}  
interface D extends A, C{}

答案是：之前那段代码没有错误，加上后面这两行之后，会产生错误！
仔细看看代码，你就会发现这是为什么了！
虽然之前那段代码是没问题的，但是读起来是不是很累呢？事实上，我们应该尽量避免这种情况的发生！

接口中的域

任何放入接口中的域都是static和final的 这个特性使得在Java SE5之前常用来创建常量组： 

public interface Months{
  int JAN = 1, FEB = 2, MAR = 3,
      APR = 4, FRI = 5, JUN = 6,
      JUL = 7, AUG = 8, SEP = 9,
      OCT = 10, NOV = 11, DEC = 12;
}

(注： Java SE5 之后，你可以使用更加灵活强大的关键字 enum —— 枚举类型)

接口中定义的域不能使“空final”的，但是可以被非常量表达式初始化，比如一个随机数。

有趣的嵌套接口

在类中嵌套一个或者多个接口是允许的，而且有的时候非常有用！
假设你在某个类里面嵌套一个接口，并且该接口在该类内部使用，那么你不能说该类实现了该接口，但是该类却拥有和实现这个接口的其他类一样的能力！
这样做的结果就是：该类不需要实现接口就获得了接口的能力，并且不带有其他类型信息（因为不能够被向上转型为接口类型）
是不是觉得很有趣？
值得注意的是：

  • 接口之间也是可以嵌套的，但是由于借口内部的元素都是public的，所以，嵌套在一个接口内部的接口自动就是public的！
  • 当实现某个接口时，并不需要实现该接口内部的接口

接口嵌套的威力不止这些，配合权限，你可以玩出更多花样！

一句话小结

接口就是一个比抽象类更加抽象的特殊的类，但是接口有许多它自己的特性！

Java正则表达式

正则表达式的定义

一个正则表达式可以是：

  • 空字符串
  • 单个字符
  • 包含在括号中的另一个正则表达式
  • 两个或多个连接起来的正则表达式
  • 由或运算符分隔的两个或者多个正则表达式
  • 由闭包运算符标记的正则表达式

每个正则表达式都是一个字符串的集合：

  • 空正则表达式表示的字符串集合为空，含有0个元素
  • 一个字符表示的字符串的集合含有1个元素，即该字符本身。
  • 一个由括号和包含在其中的正则表达式组成的正则表达式表示的字符串的集合与括号内的正则表达式相同。
  • 由两个正则表达式连接起来的正则表达式表示的字符串的集合为这两个正则表达式分别表示的字符串集合的叉乘。
  • 由或运算符连接的两个正则表达式所表示的字符串的集合为两个正则表达式分别所表示的字符串集合的并集。
  • 由一个正则表达式的闭包所表示的字符串的集合由空字符串或者将被修饰的正则表达式所表示的字符串集合重复任意次所得到的所有字符所组成。

正则表达式的描述模式

模式的描述由四种基本的操作和作为操作的字符组成。

链接操作

比如我们写出这样的表达式“AB”，它是一个包含两个字符的字符串，由A和B连接而成。

或操作

或操作表示模式中多种可能的匹配。我们在两种选择之间指定一个“或”运算符。“A|B”这个字符串表示：A或者B。连接操作的优先级高于或操作。如“AB|BCD”表示：AB或者BCD。

闭包操作

闭包操作表示将模式的部分重复任意的次数，包括零次。我们将““符号标记在需要重复的模式之后以表示闭包。”AB“表示一个A或者任意个B；”A*B“表示任意个A和一个B。

括号

括号用来改变默认的优先级顺序。例如A(B|CD)E表示：ABE或者ACDE。(AB)*表示：将(AB)连接任意多次（包括零次）。

缩略写法

一般的应用程序都在基本的规则基础上增加了各种额外规则，以力求简洁地描述实际应用中所需要的效果。从理论上看，它们都是涉及多个操作数的一系列操作的简略写法，从实际角度来看，他们都是对基本操作的拓展，以便能够写出小巧的模式。

String与StringBuilder

Java中的String类型是不可变的，String类被声明为final,所以每一个看起来会修改String值的方法都是新创建了一个新的String对象，而原来的String对象未改变。
String对象是不可变的，但是你可以给它加任意多个引用，任何指向它的引用都不能改变它的值。
Java为String对象重载了“+”、“+=”两个运算符，这也是Java中仅有的两个重载过的运算符，程序员不允许自己重载运算符。
看下面一个例子： 

String mango = "mango";

上面创建s对象的时候，创建了几个String对象？答案是4个！第一个自然是mango，然后每次运用“+”实际上都是创建了一个新的String对象。我想你应该发现问题所在了：为了新建一个String对象，居然创建了这么多中间对象！而且这些对象用过一次之后再也没用了！
显然，Java的设计者意识到了这个问题。于是有了我们熟悉的StringBuilder，它比String高效！值得注意的是，我们在代码之中可以使用String的方式来创建对象，因为编译器在编译的时候会自动的引入StringBuilder，即使我们没有使用StringBuilder。
