title: Java Interface
date: 2016-7-30 20:47:30

tag: java

在Java中，接口(interface)和内部类为我们提供了一种将接口与实现分离的更加结构化的方法。

抽象类与抽象方法

包含抽象方法的类叫做抽象类。如果一个类包含一个或者多个抽象方法，则该类必须限定为抽象的。
抽象类其实起着这样一个作用——通用接口。不同的子类可以用不同的方式表示此接口。这和我们讨论的接口是不是很像呢？这也是为什么在说接口(interface)之前要说抽象类的原因。
如果想从一个抽象类继承，并且创建该新类的对象，则必须为基类中的所有抽象方法提供方法定义。这一点与C++是一样的。如果不这么做，那么导出类也是一个抽象类，并且编译器会强制我们使用
abstract关键字来限定这个类。
创建抽象类和抽象方法非常有用，因为它们可以使累的抽象性明确起来，并且告诉用户和编译器打算怎么来使用它们。

你真的理解了接口吗？

abstract关键字允许人们在类中创建一个或者多个没有任何定义的方法——提供了接口部分，但是没有提供具体的实现，这些是此类的导出类完成的。
interface关键字使得抽象的概念更向前迈进了一步。interface关键字创建一个完全抽象的类：它根本就没有提供任何具体实现。它允许创建者定义方法和签名，但是没有任何方法体。
（注：Java 8 支持在interface中包含方法体，这是Java 8的新特性之一。更多新特性，请自行查看Java 8 release note）
一个接口表示：“所有实现了该特定接口的类看起来都像这样”。
interface不仅仅是一个极度抽象的类，它还允许我们创建一个能够被向上转型为多种基类的类型。
创建一个接口很简单，用interface关键字代替class关键字即可。例如： 

interface SampleInterface{}

要让一个类遵循某个接口（或者一组接口），需要使用implements关键字，它表示：“interface知识外貌，我现在要说明它是如何工作的”。也就是实现interface中声明的方法。例如： 

class SampleClass implements SampleInterface{}

值得注意的是：

  • 接口是隐式抽象的，不需要abstract关键字修饰
  • 接口中的方法也是隐式抽象的，不需要abstract关键字修饰
  • 接口中的方法访问权限都是public

接口支持多重继承！

我们知道，Java中类是不支持多重继承的（C++支持多继承），但是接口是完全支持多继承的。一个类想“继承”多个接口，只需要在implements关键字后面，逐个写出接口名，用“，”隔开。例如： 

interface I1{ void method1();}  
interface I2{ void method2();}
interface I3{ void method3();}  
class A implements I1, I2, I3{
  void method1(){}
  void method2(){}
  void method3(){}
}

不知道你发现没有，这样的特性使得该类可以向上转型为任何一个接口类型，因为每一个接口都是一个独立的类型！ 这样的特性十分灵活！
总结一下使用接口的原因：

  • 为了能够向上转型为多个类型以及由此带来的灵活性
  • 防止客户端程序员创建该类的对象，并且确定这只是一个接口——这一点与抽象类相同。

那么，我们应该使用接口还是抽象类呢？

如果要创建不带任何方法定义和成员变量的基类，那么就应该选择接口而不是抽象类。接口优先于抽象类。

接口的扩展

说到扩展接口，第一反应就是通过继承现有的接口。比如这样： 


interface A {}  
interface B extends A{ }

这里的extends关键字是不是很熟悉？没错，它和非接口类的继承使用同样的关键字。我们知道一般的类extends后面只能接一个基类。但是 接口可以继承多个基类！
例如： 

interface A { void methodA(); }  
interface B { void methodB(); }  
interface C extends A, B { void methodC(); }

这样一来，如果有一个类实现接口C，实际上也实现了接口A和B——C继承了A和B的方法，类实现C就必须实现其所有方法！
很显然，接口C是一个扩展的接口！

又一个麻烦

不知道你想过没有，假设有这样一种情况：覆盖、实现和重载搅在了一起，该怎么办？
举个例子： 

interface A{ void f(); }  
interface B{ int f(int i); }  
interface C{ int f(); }  
class C1 { public int f(){ return 1;}}  
class C2 implements A, B{
  public void f(){}  
  public int f(int i){ return 1;} //overload  
}   
class C3 extends C1 implements B{
  public int f(int i){ return 1;} //overload  
}  
class C4 extends C1 implements C{
  public int f() { return 1; }
}

你觉得以上程序有什么错误吗？如果加上以下两行呢？ 

class C5 extends C implements A{}  
interface D extends A, C{}

答案是：之前那段代码没有错误，加上后面这两行之后，会产生错误！
仔细看看代码，你就会发现这是为什么了！
虽然之前那段代码是没问题的，但是读起来是不是很累呢？事实上，我们应该尽量避免这种情况的发生！

接口中的域

任何放入接口中的域都是static和final的 这个特性使得在Java SE5之前常用来创建常量组： 

public interface Months{
  int JAN = 1, FEB = 2, MAR = 3,
      APR = 4, FRI = 5, JUN = 6,
      JUL = 7, AUG = 8, SEP = 9,
      OCT = 10, NOV = 11, DEC = 12;
}

(注： Java SE5 之后，你可以使用更加灵活强大的关键字 enum —— 枚举类型)

接口中定义的域不能使“空final”的，但是可以被非常量表达式初始化，比如一个随机数。

有趣的嵌套接口

在类中嵌套一个或者多个接口是允许的，而且有的时候非常有用！
假设你在某个类里面嵌套一个接口，并且该接口在该类内部使用，那么你不能说该类实现了该接口，但是该类却拥有和实现这个接口的其他类一样的能力！
这样做的结果就是：该类不需要实现接口就获得了接口的能力，并且不带有其他类型信息（因为不能够被向上转型为接口类型）
是不是觉得很有趣？
值得注意的是：

  • 接口之间也是可以嵌套的，但是由于借口内部的元素都是public的，所以，嵌套在一个接口内部的接口自动就是public的！
  • 当实现某个接口时，并不需要实现该接口内部的接口

接口嵌套的威力不止这些，配合权限，你可以玩出更多花样！

一句话小结

接口就是一个比抽象类更加抽象的特殊的类，但是接口有许多它自己的特性！

Java正则表达式

正则表达式的定义

一个正则表达式可以是：

  • 空字符串
  • 单个字符
  • 包含在括号中的另一个正则表达式
  • 两个或多个连接起来的正则表达式
  • 由或运算符分隔的两个或者多个正则表达式
  • 由闭包运算符标记的正则表达式

每个正则表达式都是一个字符串的集合：

  • 空正则表达式表示的字符串集合为空，含有0个元素
  • 一个字符表示的字符串的集合含有1个元素，即该字符本身。
  • 一个由括号和包含在其中的正则表达式组成的正则表达式表示的字符串的集合与括号内的正则表达式相同。
  • 由两个正则表达式连接起来的正则表达式表示的字符串的集合为这两个正则表达式分别表示的字符串集合的叉乘。
  • 由或运算符连接的两个正则表达式所表示的字符串的集合为两个正则表达式分别所表示的字符串集合的并集。
  • 由一个正则表达式的闭包所表示的字符串的集合由空字符串或者将被修饰的正则表达式所表示的字符串集合重复任意次所得到的所有字符所组成。

正则表达式的描述模式

模式的描述由四种基本的操作和作为操作的字符组成。

链接操作

比如我们写出这样的表达式“AB”，它是一个包含两个字符的字符串，由A和B连接而成。

或操作

或操作表示模式中多种可能的匹配。我们在两种选择之间指定一个“或”运算符。“A|B”这个字符串表示：A或者B。连接操作的优先级高于或操作。如“AB|BCD”表示：AB或者BCD。

闭包操作

闭包操作表示将模式的部分重复任意的次数，包括零次。我们将““符号标记在需要重复的模式之后以表示闭包。”AB“表示一个A或者任意个B；”A*B“表示任意个A和一个B。

括号

括号用来改变默认的优先级顺序。例如A(B|CD)E表示：ABE或者ACDE。(AB)*表示：将(AB)连接任意多次（包括零次）。

缩略写法

一般的应用程序都在基本的规则基础上增加了各种额外规则，以力求简洁地描述实际应用中所需要的效果。从理论上看，它们都是涉及多个操作数的一系列操作的简略写法，从实际角度来看，他们都是对基本操作的拓展，以便能够写出小巧的模式。

转义序列

某些字符，例如“、”、“。”、“|”、“*”、“（”、“）”都是用来构造正则表达式的元字符。我们使用反斜杠开头的转义序列来将元字符和字母表中的字符区分开来。例如：
“\“表示”\”;
“\t”表示制表符;
“\n”表示换行符;
“\s”表示空白字符。

Java中的字符串

String与StringBuilder

Java中的String类型是不可变的，String类被声明为final,所以每一个看起来会修改String值的方法都是新创建了一个新的String对象，而原来的String对象未改变。
String对象是不可变的，但是你可以给它加任意多个引用，任何指向它的引用都不能改变它的值。
Java为String对象重载了“+”、“+=”两个运算符，这也是Java中仅有的两个重载过的运算符，程序员不允许自己重载运算符。
看下面一个例子： 

String mango = "mango";  
String s = "abc" + mango + "def" + 0;

上面创建s对象的时候，创建了几个String对象？答案是4个！第一个自然是mango，然后每次运用“+”实际上都是创建了一个新的String对象。我想你应该发现问题所在了：为了新建一个String对象，居然创建了这么多中间对象！而且这些对象用过一次之后再也没用了！
显然，Java的设计者意识到了这个问题。于是有了我们熟悉的StringBuilder，它比String高效！值得注意的是，我们在代码之中可以使用String的方式来创建对象，因为编译器在编译的时候会自动的引入StringBuilder，即使我们没有使用StringBuilder。
那么直接使用StringBuilder与不使用有什么区别呢？看以下例子：

虽然编译的时候第一个方法也会使用StringBuilder优化，但是每循环一次，就会创建一个新的StringBuilder对象！但是第二个方法自始至终只有一个StringBuilder对象！
除此之外，StringBuilder对象还允许你预先为其指定大小。当你知道最终的字符串大概有多长的时候，预先指定大小可以避免多次重新分配缓冲区。所以，我们应该尽量多的使用StringBuilder而不是String！

Basic git

Git是什么

Git是一个广泛使用的分布式版本管理系统（Distributed Version Control System）
，在这类系统中，客户端并不只提取最新版本的文件快照,而是把原始的代码
仓库完整地镜像下来。这么一来,任何一处协同工作用的服务器发生故障,事后都可以用任何一个镜像出来的
本地仓库恢复。因为每一次的提取操作,实际上都是一次对代码仓库的完整备份。

Git的三种状态

分别是：已提交(committed)、已修改(modified)、已暂存(staged)。
已提交表示该文件已经被安全地保存在本地数据库中了;
已修改表示修改了某个文件,但还没有提交保存;
已暂存表示把已修改的文件放在下次提交时要保存的清单中。
由此我们看到 Git 管理项目时,文件流转的三个工作区域: Git 的本地数据目录,工作目录以及暂存区域。

Android Handler

Handler、MessageQueue和Looper的概念

Handler是Android的消息机制，其运行需要底层的MessageQueue和Looper支撑。MessageQueue即消息队列，但是它的内部存储结构并不是队列，而是采用的单链表。当然，在开发过程中，你可以创建自己的消息处理框架，比如自己实现一个MessageQueue。Looper即消息循环。MessageQueue只是一个存储消息的单元，它并不会处理消息。消息处理由Looper提供，它会无线循环的查找是否有新消息，如果有则处理，否则一致检查。值得注意的是，Looper之中还有一个ThreadLocal，这不是一个线程，它的作用是可以在每个线程中存储数据。ThreadLooper可以获取每个线程的Looper。
当我们新建一个线程时，默认是没有Looper的，如果要使用Handler，则必须为该线程创建Looper。我们熟悉的主线程为什么就可以直接使用Hanlder呢？其原因是：主线程（UI线程）即ActivityThread被创建的时候就初始化了Looper！

延伸：为什么Android不允许在子线程访问UI

这是因为Android的UI线程不是线程安全的！当多线程并发访问的时候，可能会导致UI控件处于不可预期的状态。但是，我们可以给UI控件上锁啊，这样不就可以解决这个问题吗？答案是：锁机制会让UI访问的逻辑变得复杂，会影响UI的访问效率，因为锁机制会阻塞某些线程的执行。
所以，在子线程上，如果想要更新控件，不能直接像主线程那样，你应该在子线程，通过Handler发送一个消息到主线程，通知主线程来完成这个操作。

ThreadLocal的工作原理

ThreadLocal是一个线程内部的数据存储类，通过它可以在指定的线程之中存储数据，数据存储以后，只有在指定的线程中可以获取数据，其他线程则无法获取。
ThreadLocal的奇妙之处是：虽然不同线程访问的是同一个ThreadLocal对象，但是该对像的值却是不同的！
看下面的例子： 

private ThreadLocal<Boolean> mThreadLocal = new ThreadLocal<Boolean>();
mThreadLocal.set(true);
Log.d("MainThread","MainThread.mThreadLocal = " + mThreadLocal.get());
new Thread("Thread_1"){
    @Override
    public void run(){
        mThreadLocal.set(false);
        Log.d("Thread_1","Thread_1.mThreadLocal = " + mThreadLocal.get()); 
    };
}.start();
new Thread("Thread_2"){
    @Override
    public void run(){
        Log.d("Tnread_2","Thread_2.mThreadLocal = " + mThreadLocal.get());
    };
}.start();

运行上面的代码，可以得到一下结果： 

MainThread.mThreadLocal = true
Thread_1.mThreadLocal = false
Thread_2.mThreadLocal = null

是不是感觉很神奇？！
ThreadLocal的set方法，是通过Values对象来获取当前线程中的ThreadLocal数据，Thread对象的内部有一个ThreadLocal.Values对象localValues，把数据存储在localValues对象的内部数组table中，即实现set方法。同样的，对于ThreadLocal的get方法，也是先取出当前线程的localValues这个对象，接着取出其table数组的数据，即实现了get方法。
可以看出，set\get方法都是操作当前线程的localValues对象的table数组，所以不同线程取出的值是不一样的。也就是说，ThreadLocal可以在不同线程之间互不干扰的存储和修改数据。

MessageQueue的工作原理

MessageQueue主要包含两个操作：插入和读取。读取操作会伴随着删除。插入消息到队列使用enqueueMessage方法，读取使用next方法，读取一条消息并将其从消息队列中删除。enqueueMessage方法主要就是单链表的插入操作，而next方法是一个无线循环的方法，如果队列中没有消息，则阻塞，当有消息到来时，next方法会返回这条消息并且将其从单链表中移除。

Looper的工作原理

Looper就是不停的查看MessageQueue中是否有新消息，有消息就立刻处理，无消息则阻塞。Looper的构造方法中会新创建一个MessageQueue对象，然后获得当前线程的对象。如何为当前线程创建Looper对象呢？直接调用Looper.prepare()方法即可。接着，调用Looper.loop()方法开启循环。对于主线程，Looper提供了专门的preoareMainLooper方法，其本质也是prepare实现的。主线程的Looper比较特殊，Looper提供了getMainLooper方法，可以在任何地方获取主线程的Looper对象。Looper退出有两种，一种是quit，一种是quitSafely。在线程中手动创建了Looper的话，应该在不需要的时候推出Looper，否则线程一直处于等待状态。
Looper的loop方法是一个死循环，唯一跳出循环的条件是MessageQueue的next方法返回了null。Looper处理消息是通过发送消息的哪个Hanlder对象实现的,即msg.target.diapatchMessage(msg)方法,msg.target就是发送这条消息的Handler对象。所以，Handler给MessageQueue发送消息，被Looper检测到，又交给Handler自身去执行了！只不过，handler的diapatchMessage方法是在创建Handler对象的线程里面执行的！

Handler的工作原理

Handler的主要工作就是发送消息和接收并处理消息。消息的发送通过post的一系列方法和send的一系列方法实现，而post的一系列方法是通过send的一系列方法实现的。Handler发送消息的过程仅仅是向消息队列插入一条消息，MessageQueue的next方法返回这个消息给Looper，Looper接着交给Handler对象处理，Handler通过dispatchMessage方法处理该消息，该方法的实现如下： 

public void dispatchMessage(Message msg){
    if(msg.callback != null){handleCallback(msg);}
    else{
        if(mCallback != null){
            if(mCallback.handleMessage(msg)){return;}
        }
        handleMessage(msg);
    }
}

Message的callback是一个Runnable对象，实际上就是Handler的post方法所传递的Runnable参数。handleCallbck很简单，就是运行这个Runnable对象msg.callback。
mCallback是一个Callback类型的接口，其定义如下： 

public interface Callback{    public boolean handleMessage(Message msg);}

dispatchMessage方法首先判断mCallback是否为空，否则调用mCallback的handleMessage方法，为空则直接调用handleMessage方法。
Callback还有一个作用，那就是用来构造Handler对象，尤其是你的Handler不需要派生子类的时候。我们通常都是派生一个子类重写其handleMessage方法。如果不想派生子类，则通过new Handler(Callback callback)来构造Handler。
除此之外，Handler还有一个构造方法，即通过Looper对象：public Handler(Looper looper){}

主线程的消息循环

主线程的消息循环和子线程类似，只不过在主线程的入口方法main里面，是使用Looper.prepareMainLooper方法来创建主线程Looper和MessageQueue的。

BaseAdapter中使用ViewHolder

BaseAdapter在Android开发中用的很多，尤其是在给ListView做适配器的时候。
当ListView中含有许多项目的时候，Android系统会有一个缓存策略，而ViewHolder就是充分利用缓存以提高性能的典型。
在如今流行的RecyclerView中也可以看到它。那么在BaseAdapter如何使用ViewHolder呢?
Talk is cheap, show me the code

不使用ViewHolder的例子：

private class MyAdapter extends BaseAdapter{
    private List<Map<String, String>> items;
    public MyAdapter(List<Map<String, String>> list){
        this.items = list;
    }
    @Override
    public int getCount() {
        return items.size();
    }
    @Override
    public Object getItem(int i) {
        return items.get(i);
    }
    @Override
    public long getItemId(int i) {
        return i;
    }
    @Override
    public View getView(int position, View view, ViewGroup viewGroup) {
        view  = getLayoutInflater.inflate(R.layout.item,null);
        TextView text = (TextView) view.findViewById(R.id.text);
        Button button = (Button) view.findViewById(R.id.button);
        text.setText("我是第"+position+"项");
        button.setText("我是第"+position+"个按钮");
        return view;
    }
}

使用ViewHolder的例子：

private class MyAdapter extends BaseAdapter{
    private List<Map<String, String>> items;
    public MyAdapter(List<Map<String, String>> list){
            this.items = list;
    }
    @Override
    public int getCount() {
        return items.size();
    }
    @Override
    public Object getItem(int i) {
        return items.get(i);
    }
    @Override
    public long getItemId(int i) {
        return i;
    }
    @Override
    public View getView(int position, View view, ViewGroup viewGroup) {
        ViewHolder viewHolder = new ViewHolder();
        if(view == null){
            view  = getLayoutInflater.inflate(R.layout.item,null);
            viewHolder.text = (TextView) view.findViewById(R.id.text);
            viewHolder.button = (Button) view.findViewById(R.id.button);
            view.setTag(viewholder);
        }else{
            viewHolder = view.getTag();
        }
        viewHolder.text.setText("我是第"+position+"项");
        viewHolder.button.setText("我是第"+position+"个按钮");
        return view;
        }
    private final class ViewHolder{
        public TextView text;
        public Button button;
    }
}

就是这么简单！希望能对初学者有点帮助！
